## 2. Customer Support Login Lockout — RESOLVED

### 2.1 Symptom (carried from v7 §4.2)
`support.test@high6.dev` logged in successfully but could not reach the dashboard:
```
GET /admin/users/me → 400 "Access denied: missing permission user:read"
```

### 2.2 Investigation findings
- **DB-confirmed (queried directly, not seed logs):** 5 of 8 roles lack `user:read` — Customer
  Support, Operations Manager, Order & Fulfillment Staff, Marketing, Catalog/Product Manager.
  The 3 that have it (Store Owner, Read-Only/Auditor, Developer/Platform Support) have it
  coincidentally, because their business scope happens to touch users — not because a "baseline
  dashboard bootstrap" permission tier was ever designed. This is why Store Owner-only testing
  masked the bug.
- **Root cause is in our guard, not Medusa core.** The 400 is generated by `rbac-guard.ts`, which
  maps `GET /admin/users/me` → resource `user`, action `read`, and runs a policy check requiring
  `user:read`. Medusa core deliberately attaches **no** policies to this route
  (`node_modules/@medusajs/medusa/dist/api/admin/users/middlewares.js:69-75`) — upstream design
  is that any authenticated admin can read their own identity. Our guard is stricter than the
  framework it wraps.
- `rbac-guard.ts:38-39` already documents the intended behavior — "/me/* endpoints — accessible
  to all authenticated users (returns own data)" — but the code never implemented that intent;
  it ran the policy check anyway.
- A leftover **"Test Role" with 0 policies** was found in the DB during this audit (likely from
  the v6 Store-Owner-can-create-roles walkthrough) — not touched, flagged for cleanup.

### 2.3 Fix — guard self-endpoint bypass (not per-role policy grants)
Considered and rejected: adding `user:read` to the 5 missing roles. This was rejected because
`user:read` also grants `GET /admin/users` (full team roster) and would surface Settings→Users
for roles whose v6 scope explicitly excludes Settings (Operations Manager, Order & Fulfillment
Staff) — trading a login bug for a scope violation.

**Fix implemented:** `rbac-guard.ts` now skips the policy/permission check specifically for
self-endpoints — `GET /admin/users/me` and `GET /admin/me/*` (this also covers
`/admin/me/permissions`, the sidebar-filter endpoint from v7 §2.4/§3, which had the identical
exposure). Authentication (session/JWT) is still fully enforced; only the authorization check is
bypassed, and only for these two route patterns.

**Why this is scalable:** no role's policy list was modified. All 8 roles' policy counts are
unchanged. Any future role gets past dashboard bootstrap automatically, with zero code changes
required per role.

### 2.4 Verification
- All 5 previously-locked-out roles confirmed able to reach the dashboard (browser-verified for
  `support.test`; `owner.test` re-checked as unaffected/unchanged).
- Unauthenticated requests to the same routes still rejected — auth bypass was scoped to
  authorization only.
- 38/38 tests passing, 0 new type errors.

## 3. Sidebar Nav Showing Out-of-Permission Items — RESOLVED

### 3.1 Symptom
After the §2 fix unblocked login, Customer Support's sidebar displayed nav items it has no
permission for. Clicking them correctly denied access (route guard/policy enforcement was never
the problem) — only the sidebar's own item filtering was wrong.

This was a **recurrence**, not a new bug class: a matching symptom (Store Owner seeing Developer
nav items) had previously appeared "fixed." Its reappearance on a different role meant the prior
fix was treated as unproven rather than a trusted precedent.

### 3.2 Root cause
The prior Store Owner/Developer fix was a **text-matching hack** — it hid a nav section by
matching the literal string "Developer." This masked the symptom for one specific section on one
specific role and did not generalize to any other role or section, and would not have survived
i18n either.

### 3.3 Fix — structural, permission-driven, not role-specific
Two changes, both generic:
1. **Section collapse rule:** any sidebar group whose nav links are all hidden is now collapsed
   entirely, via a structural walk-up from each hidden link (checks whether the containing
   subtree contains only hidden links). Replaces the old "Developer" string match. Works for
   "General," "Developer," and any future section, and no longer depends on the English word
   "Developer."
2. **New `ACTION_ROUTE_PERMISSIONS` map:** write-action links (`…/create`,
   `/products/import|export`, `/settings/users/invite`, etc.) are now hidden when the user lacks
   the corresponding write permission. Verified these render as static anchors in the dashboard
   bundle, so CSS-based hiding is reliable for this class of link.

### 3.4 Known boundary — not fixed this session
Row-action menus (e.g. Delete on a customer row), dynamic edit routes (`/…/:id/edit`), and
order-detail action flows are buttons/dynamic routes, not static anchors — they remain visible
and deny-on-click. Direct URL-typing to a create route still opens the form modal, but submit is
denied by the backend (the guard remains the actual enforcement layer; this whole fix is UX
only). **Logged as follow-up** — needs a per-page pass, not a global rule like the two above.

### 3.5 Verification
- `support.test`: Settings shows no "General" heading (Profile section only); Products page has
  no Create button, Import/Export menu absent; Orders and Customers still show Create (role has
  those permissions).
- `owner.test`: unchanged from prior checklist (Developer section gone, Roles visible) — confirms
  the new structural rule didn't regress Store Owner's correct-and-already-working view.
- 38/38 tests passing.

## 4. Customer Support — Current Interface Checklist

| Area | Can | Cannot |
|---|---|---|
| Orders / Draft Orders | View, create, edit (addresses/email), cancel, notes | ⚠️ Returns, Claims, Exchanges, Order Edits, Refunds, Captures — separate resources (`return`, `order_claim`, `order_exchange`, `order_change`, `refund`, `capture`) not granted → deny on click inside an order |
| Customers | View, create, edit | Delete (row menu denies), Customer Groups (hidden) |
| Products | View list/detail only | Create/Import/Export (now hidden), edit/delete (deny on click) |
| Inventory | View items only | Create (hidden), adjust levels (deny), Reservations (hidden) |
| Settings | Profile only | Everything else (hidden) |
| Everything else | — | Promotions, Campaigns, Price Lists, Categories, Collections, Gift Cards — all hidden |

**Role-definition gap identified (not fixed):** v6 intended Customer Support to handle "refund
requests," but the role has no `refund:create` — it cannot actually process a refund. This is a
policy-list/role-scope decision, not a code bug. Changing policy lists was out of scope this
session; logged as an open item below.
